Module Development Guidelines

CSS Structure

  • Use BEM naming: .yourmodule, .yourmodule__element, .yourmodule--modifier
  • Root container: wrap with class="tonika-module yourmodule yourmodule--card"
  • Use global components: .tonika-btn, .tonika-input, .tonika-select
  • Use CSS variables: e.g. var(--color-bg-primary), var(--spacing-md)

JavaScript Integration

  • Mount your module into #demo-mount (or your target element)
  • Generate HTML with proper BEM classes
  • Provide a destroy() for cleanup
  • Prefer the event system (see TonikaEmitter below)

🧃 Soundonika Audio Integration (local, emitter-based)

Soundonika is part of Tonika core. Initialize the engine and listen to status events. Then schedule sounds.

// Ensure core scripts are loaded (see <head>)
const ac = new (window.AudioContext || window.webkitAudioContext)();
const engine = new Tonika.SoundonikaEngine(ac, {
  sampleBasePath: './samples',
  volume: 0.9
});

// Optional: status events (loading → ready → error)
const off = engine.addEventListener('status', e => console.log('[sound]', e.detail));

// Init then schedule
engine.init().then(() => {
  const t = ac.currentTime + 0.02; // small schedule-ahead
  engine.scheduleSound(t, 'kick', 1.0);
});

Supported default sound types: 'kick', 'snare', 'hihat', 'perc', 'clap'. You can override with setSampleMappings().

Sample path strategy:

  • Local samples: keep packs under /samples/ and update sample-index.json as needed.
  • Custom mappings: call engine.setSampleMappings({...}) to map friendly names to file paths.
  • Missing sample fallback: engine will play a click tone if a mapping/path isn’t loaded.

Events you can rely on (core)

  • Clavonika: ui:noteon, ui:noteoff, status (factory returns an instance with on/off).
  • Jackonika: midi:noteon, midi:noteoff, midi:devicechange, status (singleton).
  • Soundonika: status (loading/progress/ready/error).
